# Flet製 謎解き脱出ゲーム 制作記録・技術解説

## 1. この資料の目的

この資料は、PythonとFletを使って制作した謎解き脱出ゲーム「放課後の情報室からの脱出」について、制作の流れ、使った技術、技術を選んだ理由、GitHub Pagesで公開するまでの手順をまとめたものです。

ゼミで説明しやすいように、単なる作業メモではなく、初学者が「なぜそうしたのか」を理解できる形で整理しています。

Pythonは、読みやすい文法が特徴のプログラミング言語です。

今回Pythonを使った理由は以下です。

- 初学者でもコードを読みやすい

このアプリでは、例えば以下のような状態をPythonの辞書で管理しています。

game_state = {

これは「現在プレイヤーが何を持っているか」「パソコンを解除したか」「ロッカーを開けたか」「クリアしたか」などを記録するためのものです。

## 6. Fletとは何か

Fletは、PythonでGUIアプリやWebアプリを作るためのフレームワークです。

通常、Webアプリを作る場合はHTML、CSS、JavaScriptを使うことが多いです。しかしFletを使うと、Pythonのコードだけでボタン、テキスト、入力欄、ダイアログなどを作れます。

今回Fletを使った理由は以下です。

- Pythonだけで画面付きアプリを作れる
- `Text`、`Button`、`TextField`、`AlertDialog` などの部品が用意されている
- デスクトップアプリとしてもWebアプリとしても動かせる
- 画面遷移、ボタン操作、入力フォーム、状態管理の学習に向いている
- ゲームエンジンを使わずに、学習用の脱出ゲームを実装できる

例えば、Fletでは以下のようにボタンを作れます。

```python
ft.ElevatedButton(
    "ゲーム開始",
    icon=ft.Icons.PLAY_ARROW,
    on_click=lambda event: show_game(),
)
```

このコードは、「ゲーム開始」というボタンを作り、クリックされたら `show_game()` という関数を呼び出す、という意味です。

## 7. uvとは何か

`uv` は、Pythonの仮想環境作成やパッケージインストールを高速に行うためのツールです。

今回の環境では `pip` が最初から使えなかったため、`uv` を使って仮想環境を作り、Fletをインストールしました。

実際に使った考え方は以下です。

```bash
uv venv .venv
uv pip install -r requirements.txt
```

`.venv` は仮想環境です。仮想環境とは、プロジェクト専用のPython環境です。

仮想環境を使う理由は以下です。

- 他のPythonプロジェクトと依存関係が混ざらない
- このゲームに必要なパッケージだけを管理できる
- 後から環境を再現しやすい

## 10. ゲーム内容を難しくした理由

最初の実装では、ヒントを得たらすぐにそのまま入力するだけの流れでした。

例えば、最初のバージョンでは「時計が14:25を指している」というヒントを得たら、すぐに `1425` を入力するような構造でした。これは処理としては分かりやすいですが、謎解きとしては単調でした。

そこで改善版では、1つのヒントだけでは答えが出ないようにしました。

現在の謎解きの考え方は以下です。

1. 机で「係メモ」を見つける
2. 黒板で「座席表」と止まった時計を確認する
3. 係メモと座席表を組み合わせて、パソコンのパスワードを解く
4. パソコンから「色ログ」を得る
5. 係メモ、座席表、色ログを組み合わせてロッカー番号を解く
6. ロッカーからドアカードを得る
7. プリンタから最終メモを得る
8. 時刻、ロッカー番号、文字数を組み合わせてドアを開ける

このように、「情報を集める」「整理する」「組み合わせる」という流れを作ることで、単なる入力作業ではなく謎解きらしくしました。

## 11. 謎解きの具体例

### 11.1 パソコンの謎

係メモには以下の情報があります。

```text
PC係: B2
図書係: A3
ロッカー係: C1
```

黒板の座席表には以下の情報があります。

```text
A1=HTML / A2=Python / A3=Flet
B1=Django / B2=Linux / B3=CSS
C1=青・赤・緑 / C2=Scratch / C3=JavaScript
```

PC係は `B2` です。

座席表で `B2` を見ると `Linux` と書かれています。

したがって、パソコンのパスワードは以下です。

```text
linux
```

### 11.2 本棚の謎

図書係は `A3` です。

座席表で `A3` を見ると `Flet` と書かれています。

したがって、本棚で選ぶべき本は以下です。

```text
flet
```

正解すると小さな鍵を入手します。

### 11.3 ロッカーの謎

ロッカー係は `C1` です。

座席表で `C1` を見ると、以下の色順が書かれています。

```text
青・赤・緑
```

パソコンを解除すると、色ログが表示されます。

```text
青=3
赤=7
緑=9
```

色順に数字を並べると以下になります。

```text
379
```

これがロッカーの暗証番号です。

### 11.4 最後のドアの謎

プリンタから得られる最終メモには以下の内容があります。

```text
最後の暗証番号は、時刻の下2桁 + ロッカー番号 + 図書係の答えの文字数。
```

黒板横の時計は `17:40` で止まっています。

時刻の下2桁は以下です。

```text
40
```

ロッカー番号は以下です。

```text
379
```

図書係の答えは `Flet` なので、文字数は以下です。

```text
4
```

これらをつなげると、最後の暗証番号は以下になります。

```text
403794
```

## 12. 状態管理の考え方

ゲームでは「今どこまで進んだか」を覚えておく必要があります。

例えば、以下のような状態があります。

- 机を調べたか
- 係メモを持っているか
- パソコンを解除したか
- ロッカーを開けたか
- ドアカードを持っているか
- クリア済みか

これらを `game_state` という辞書でまとめて管理しています。

状態管理を行うことで、以下のような処理ができます。

- すでに取ったアイテムを二重に取らせない
- 必要なアイテムがない場合は先に進めない
- クリア後は通常の調査を止める
- 行動回数を数える
- メッセージ欄に現在の状況を表示する

## 18. FletアプリをGitHub Pagesで公開する難しさ

GitHub Pagesは基本的に静的サイト向けです。

一方、今回のアプリはPython/Fletで作っています。そのまま `main.py` をGitHub Pagesに置いても、ブラウザはPythonファイルを直接実行できません。

そのため、FletのWebビルド機能を使って、ブラウザで動く静的ファイルへ変換しました。

実行したビルドの考え方は以下です。

```bash
.venv/bin/flet build web --base-url /escape-game-flet/ --route-url-strategy hash
```

重要なのは `--base-url /escape-game-flet/` です。

GitHub Pagesでは、今回のアプリは以下のようなサブディレクトリで配信されます。

```text
https://kiso-12.github.io/escape-game-flet/
```

そのため、アプリ内のJavaScriptや画像などの読み込みパスも `/escape-game-flet/` を基準にする必要があります。これを指定しないと、公開後にファイルが見つからないことがあります。

## 19. GitHub Actionsとは何か

GitHub Actionsは、GitHub上で自動処理を実行できる仕組みです。

今回の用途では、GitHub Pagesへファイルをデプロイするために使いました。

設定ファイルは以下です。

```text
.github/workflows/deploy-pages.yml
```

このファイルには、「どのタイミングで」「どの処理を」「どの権限で」実行するかを書きます。

今回のワークフローでは、主に以下の処理をしています。

1. `gh-pages` ブランチをチェックアウトする
2. GitHub Pages用の設定を行う
3. 静的ファイルをPages用アーティファクトとしてアップロードする
4. GitHub Pagesへデプロイする

今回は、ソースコードと公開用ファイルを分けています。

| ブランチ | 役割 |

## 22. 作業中に起きた問題と対処

### 22.1 pythonコマンドがなかった

環境によっては `python` コマンドがなく、`python3` しか使えませんでした。

対処として、構文チェックなどでは以下を使いました。

```bash
python3 -m py_compile main.py
```

ただし、仮想環境を作成した後は以下で実行できます。

```bash
.venv/bin/python main.py
```

### 22.2 pipがなかった

この環境では `pip` や `ensurepip` が使えませんでした。

そのため、`uv` を使ってパッケージをインストールしました。

```bash
uv pip install -r requirements.txt
```

### 22.3 日本語が文字化けした

最初にFletデスクトップアプリを起動したとき、日本語フォントが見つからず文字化けしました。

対処として、Windows側にあったMeiryoフォントをユーザー領域にコピーし、Flet側で明示的に指定しました。

```python
page.theme = ft.Theme(font_family="Meiryo")
```

### 22.4 FletのAPI差分でエラーが出た

Flet 0.85.3では、古い書き方や別バージョンの書き方が一部使えませんでした。

例えば以下のような修正を行いました。

| 変更前 | 変更後 |
| --- | --- |
| `ft.app(target=main)` | `ft.run(main)` |
| `ft.alignment.top_left` | `ft.Alignment(-1, -1)` |
| `ft.padding.symmetric(...)` | `ft.Padding(...)` |
| `ft.border.all(...)` | `ft.Border(...)` と `ft.BorderSide(...)` |
| `page.dialog = dialog` | `page.show_dialog(dialog)` |
| `page.snack_bar = ...` | `page.show_dialog(ft.SnackBar(...))` |

この経験から、ライブラリはバージョンによって書き方が変わることが分かります。

### 22.5 GitHub CLIの認証が切れていた

最初、`gh auth status` を実行すると、GitHubのトークンが無効でした。

そのため、以下で再ログインしました。

```bash
gh auth login -h github.com -p https -w
```

ブラウザ自動起動ができなかったため、GitHubのデバイス認証ページに手動でコードを入力しました。

### 22.6 GitHub Pagesの反映に時間がかかった

最初は `gh-pages` ブランチを直接Pagesに設定しましたが、Pagesの状態がしばらく `building` のままでした。

そこでGitHub Actions方式に切り替えました。

最終的にはPages設定が以下になりました。

```text
build_type: workflow
status: built
```

## 26. まとめ

今回の制作では、PythonとFletを使って、ローカルで動く謎解き脱出ゲームを作成しました。

最初はシンプルなボタン中心のゲームでしたが、途中でデザインと謎解きの内容を改善し、情報室らしい世界観と、複数の手がかりを組み合わせるゲーム性を追加しました。

さらに、Flet Webビルド、GitHub、GitHub Pages、GitHub Actionsを使い、ブラウザからアクセスできる形で公開しました。

このプロジェクトは、Pythonの基礎だけでなく、アプリ制作、UI設計、状態管理、GitHub公開までを一通り体験できる学習用題材になっています。
